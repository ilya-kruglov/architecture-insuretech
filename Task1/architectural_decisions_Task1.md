# Задание 1. Проектирование технологической архитектуры (to‑be)

Я спроектировал технологическую архитектуру «to be» для InsureTech, которая отвечает требованиям:

- Доступность 99,9% (RTO 45 мин, RPO 15 мин).
- Географически распределённая обработка для равного времени отклика по всем регионам РФ.
- Горизонтальное масштабирование подов и кластеров.
- Отказоустойчивая БД с репликацией и автоматическим переключением.

## Ключевые решения, отражённые на схеме

### 1. Два независимых кластера Kubernetes в разных регионах (например, «Запад» – Москва, «Восток» – Новосибирск)

- Каждый кластер размещён в своём ЦОД (или зоне доступности облака).
- Внутри каждого кластера – все микросервисы (`core-app`, `client-info`, `ins-product-aggregator`, `ins-comp-settlement`).

**Почему независимые кластеры, а не один растянутый?**
- Растянутый кластер требует низкой задержки между нодами (<5 мс), что невозможно между регионами РФ.
- Независимые кластеры дают полную географическую изоляцию сбоев и позволяют направлять пользователя в ближайшую точку присутствия.

### 2. Глобальный балансировщик нагрузки (GSLB) перед кластерами

- Реализация: GeoDNS или облачный Global Load Balancer (например, Yandex Cloud Network Load Balancer с гео-маршрутизацией).
- Направляет пользователя на ближайший к нему регион на основе GeoIP.
- Проверяет **health check** каждого кластера (эндпоинт `/health`). При отказе одного региона трафик перенаправляется на другой.

### 3. Фейловер-стратегия: Active-Active для чтения, Active-Passive для записи

- Все регионы активны для чтения (обрабатывают GET‑запросы, поиск, просмотр продуктов).
- Запись (оформление страховки, изменение профиля) направляется GSLB в **регион‑мастер БД** (запад).
- При падении мастера вручную или автоматически (через менеджер кластера БД) повышается реплика до мастера.
- **RTO 45 мин** достигается за счёт автоматизированного переключения (например, Patroni или Yandex Managed Service).

### 4. Конфигурация базы данных – PostgreSQL

- **Мастер** в западном регионе, **асинхронная потоковая реплика** в восточном.
- **Резервное копирование**: ежедневные полные бэкапы + WAL‑архивация в объектное хранилище (S3).
- **RPO 15 мин** обеспечивается частотой WAL‑сброса (каждые 5–10 мин).
- **Шардирование не требуется** – общий объём 50 ГБ легко обслуживается одним мастером.

### 5. Масштабирование внутри кластера

- **HPA** (Horizontal Pod Autoscaler) – по CPU, RPS и кастомным метрикам (например, latency).
- **Cluster Autoscaler** – добавление/удаление нод при нехватке ресурсов.
- Размещение нод кластера в нескольких зонах доступности внутри региона (если поддерживается облаком).

### 6. CDN для статики

- Ускоряет загрузку HTML/CSS/JS и изображений, снижает нагрузку на бэкенд.
