# Анализ и обоснование перехода с REST на GraphQL (client‑info)

## Исходная проблема

Сервис `client-info` хранит до 500 атрибутов клиента. В REST API были созданы отдельные ресурсы:
- `/clients/{id}` – основная информация о клиенте
- `/clients/{id}/documents` – документы
- `/clients/{id}/relatives` – родственники

При этом в рамках одного сценария (например, оформление страховки) требуются разные наборы данных. Клиенту приходится делать несколько последовательных запросов, что кратно увеличивает RPS и нагрузку на сервис. С другой стороны, отдавать все 500 полей одним ресурсом невозможно из‑за избыточного объёма данных.

## Решение – GraphQL

GraphQL позволяет клиенту запрашивать ровно те поля, которые ему нужны, и получать все необходимые данные за один запрос.

### Спроектированная схема

- Сущность `Client` содержит базовые поля (`id`, `name`, `age`) и ссылки на связанные объекты (`documents`, `relatives`).
- В корневом `Query` определён запрос `client(id: ID!)`, который возвращает клиента с возможностью вложенного запроса документов и родственников.
- Для обратной совместимости добавлены отдельные запросы `clientDocuments` и `clientRelatives` (они покрывают старые REST‑эндпоинты).

### Как решаются проблемы

| Проблема | Решение GraphQL |
|----------|----------------|
| **Multiple round‑trips** | Вместо 3 запросов (`/clients`, `/documents`, `/relatives`) – один `client(id)` с вложенными полями. |
| **Over‑fetching** (передача лишних данных) | Клиент сам выбирает нужные поля, например, только `id` и `name` клиента. |
| **Under‑fetching** (недостаточно данных за раз) | Можно дозапросить недостающие данные тем же запросом, добавив нужное поле. |
| **Высокая вариативность запросов** | Разные сценарии используют разные наборы полей без изменения API. |
| **Расширяемость** | Добавление новых типов (например, `addresses`) не ломает старые запросы. |

## Вывод

GraphQL для `client-info`:
- Снижает RPS за счёт объединения запросов.
- Уменьшает объём передаваемых данных (нет over‑fetching).
- Даёт гибкость разным потребителям (веб-приложение, core-app).
- Позволяет легко расширять модель данных в будущем.

Таким образом, переход на GraphQL – оптимальное решение для высоконагруженного сервиса с большим и вариативным набором атрибутов клиента.
