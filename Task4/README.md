# Task 4. Проектирование продажи ОСАГО (to‑be архитектура)

## Контекст

В рамках развития InsureTech добавляется новый продукт – онлайн‑оформление ОСАГО. 

Требования бизнеса:
- Клиент заполняет заявку на автомобиль.
- Сервис отправляет запрос во все доступные страховые компании (10 систем) и получает предложения.
- Предложения должны отображаться в реальном времени, как только ответ пришёл (максимальное время ожидания – 60 секунд).
- Пиковая нагрузка – 2500 одновременных пользователей, создающих заявки.

## Ключевые архитектурные решения

### 1. Выделение отдельного сервиса `osago-aggregator`

- **Функция**: отправка заявок в страховые компании, опрос решений, агрегация предложений.
- **Хранилище данных** – своя БД (`osago-db`, PostgreSQL/Redis) для хранения статусов заявок, correlation‑id и полученных ответов. Это необходимо для отслеживания состояния каждой заявки и обеспечения идемпотентности.
- **API для core‑app** – **асинхронный через Kafka**. `core-app` публикует событие `OsagoRequestCreated`, `osago-aggregator` подписывается на него, а полученные предложения публикуются обратно как `osago_quotes`.

### 2. Интеграция между сервисами

| От кого | Кому | Протокол | Назначение |
|---------|------|----------|-------------|
| `web` → `core-app` | REST + WebSocket/SSE | Создание заявки ОСАГО + стриминг предложений в реальном времени |
| `core-app` → `osago-aggregator` | Kafka (osago_requests) | Асинхронная передача заявки |
| `osago-aggregator` → страховые компании | REST (polling до 60 сек) | Запрос и получение предложений |
| `osago-aggregator` → `core-app` | Kafka (osago_quotes) | Публикация полученных предложений |
| `core-app` → `web` | WebSocket | Трансляция предложений клиенту по мере поступления |

### 3. Паттерны отказоустойчивости

На основе предоставленной библиотеки `Обозначения паттернов отказоустойчивости.drawio.xml` на схеме отмечены следующие паттерны:

| Паттерн | Где применяется | Обоснование |
|---------|----------------|--------------|
| **Timeout 60 сек** | Вызовы `osago-aggregator` → страховые компании | Бизнес‑требование: максимальное ожидание ответа – 60 секунд |
| **Retry** | Вызовы `osago-aggregator` → страховые компании | Экспоненциальные повторные попытки при временных сбоях (5xx, network) |
| **Circuit Breaker** | По каждому страховщику | Предотвращает повторные вызовы к недоступной компании |
| **Rate Limiter** | Вход в `osago-aggregator` (со стороны Kafka) | Защита от пиковой нагрузки 2500 заявок, контроль количества одновременно обрабатываемых запросов |

### 4. Учёт множественных экземпляров сервисов

- Все сервисы (`core-app`, `osago-aggregator` и др.) деплоятся в Kubernetes с несколькими репликами.
- Kafka гарантирует, что каждое событие обрабатывается ровно одним экземпляром подписчика (группа потребителей).
- WebSocket‑соединения могут быть привязаны к конкретному экземпляру `core-app` (например, через session affinity или использование отдельного хранилища сессий).
- Состояние Circuit Breaker и Rate Limiter может быть централизованным (Redis + Resilience4j), чтобы разделяться между репликами.

## Результат на диаграмме C4

В файле `InsureTech_C4_container_diagram_to_be_osago.drawio` отражены:
- Новый сервис `osago-aggregator` и его БД `osago-db`.
- Все интеграции (синхронные и асинхронные) с обозначением протоколов.
- Иконки паттернов отказоустойчивости над соответствующими стрелками.
- Сохранены все существующие компоненты и связи (продукты, тарифы, взаиморасчёты и т.д.).

## Заключение

Предложенная архитектура обеспечивает:
- Реальное время (real‑time) отображения предложений благодаря WebSocket и асинхронной обработке через Kafka.
- Масштабируемость под 2500 одновременных пользователей (Kafka + HPA).
- Отказоустойчивость за счёт паттернов Timeout, Retry, Circuit Breaker и Rate Limiter.
- Слабое связывание компонентов (event‑driven подход).
