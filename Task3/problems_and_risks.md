# Проблемы и риски текущей архитектуры InsureTech (as-is) при планируемом росте нагрузки

## Контекст
- Сервис `ins-product-aggregator` синхронно опрашивает API **5 страховых компаний** (скоро станет 10) и возвращает агрегированный список продуктов.
- `core-app` запрашивает данные у `ins-product-aggregator` каждые **15 минут** (синхронно, REST).
- `ins-comp-settlement` запрашивает данные у `ins-product-aggregator` раз в **сутки** (синхронно).
- `ins-comp-settlement` раз в сутки запрашивает у `core-app` **все оформленные страховки** (синхронно, REST).
- Все сервисы хранят локальные копии данных о продуктах/тарифах.

## Выявленные проблемы и риски

### 1. Синхронные вызовы к `ins-product-aggregator` создают узкое место
- **Риск**: При росте числа страховых компаний (5 -> 10) и увеличении RPS общее время ответа агрегатора линейно растёт. Это приводит к тайм-аутам и ошибкам в `core-app` и `ins-comp-settlement`.
- **Риск**: Синхронные вызовы блокируют потоки обработки, снижая пропускную способность обоих сервисов.
- **Риск**: При недоступности одной из страховых компаний вся операция может завершиться ошибкой (отсутствует fallback).

### 2. Высокая частота обновления локальных копий (`core-app` раз в 15 минут)
- **Риск**: При 10 компаниях и высоком RPS запросов к `core-app` (поиск, просмотр) локальная копия может устаревать. Пользователи видят неактуальные тарифы.
- **Риск**: Синхронные массовые вызовы создают пиковую нагрузку на `ins-product-aggregator` и API страховщиков.

### 3. `ins-comp-settlement` раз в сутки запрашивает все оформленные страховки из `core-app`
- **Риск**: При росте количества заявок объём передаваемых данных увеличивается, синхронный REST‑запрос может занимать часы, вызывая тайм-ауты и повторные попытки.
- **Риск**: Блокировка таблиц или высокая нагрузка на БД `core-app` во время выгрузки.

### 4. Отсутствие асинхронного обмена и буферизации
- **Риск**: Любой сбой в сети или задержка внешнего API немедленно отражается на доступности собственных сервисов (каскадные отказы).
- **Риск**: Невозможность масштабировать обработку событий независимо.

### 5. Проблемы согласованности данных
- **Риск**: Локальные реплики могут отличаться, так как обновляются по разному расписанию. Например, `ins-comp-settlement` работает с суточной задержкой.
- **Риск**: При сбоях синхронизации нет автоматического восстановления (кроме следующего планового запроса).

### 6. Отсутствие мониторинга и автоматического восстановления
- **Риск**: При ошибках интеграции (например, тайм-аут у страховщика) нет ретраев с экспоненциальной задержкой или circuit breaker.

### 7. Сложность добавления новых подписчиков на данные о продуктах
- **Риск**: Если в будущем потребуется ещё один сервис (аналитика, рекомендации), придётся добавлять синхронные вызовы или менять архитектуру.

## Предлагаемые направления изменений (to‑be)
1. Перевести обновление локальных копий данных о продуктах на **Event-Driven** с использованием брокера сообщений (Kafka).
2. `ins-product-aggregator` будет публиковать события об изменении тарифов/продуктов, а `core-app` и `ins-comp-settlement` – подписываться на них.
3. Для обеспечения надёжной доставки событий внедрить паттерн **Transactional Outbox** в сервисе-источнике.
4. Заменить синхронный запрос `ins-comp-settlement` -> `core-app` на асинхронную рассылку событий (например, по окончании дня `core-app` публикует событие `InsuranceIssued`).
5. Внедрить **Idempotent Consumer** и **Circuit Breaker** для внешних вызовов.
6. Использовать Kafka для буферизации и репликации событий между сервисами.

**Это позволит:**
- Уменьшить время отклика синхронных операций.
- Повысить отказоустойчивость и масштабируемость.
- Гарантировать согласованность реплик в конечном счёте (eventual consistency).
- Легко добавлять новых подписчиков.
